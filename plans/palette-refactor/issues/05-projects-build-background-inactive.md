## TLDR

Update `projects-build` to derive `backgroundInactive` using `familyname-dark` instead of `dark-familyname`.

## What to build

In `tools/term/zsh/config/theming/projects-build`, update the jq expression that computes the `backgroundInactive` color name.

Current logic: strip the trailing shade suffix from the background color name, then prepend `"dark-"` to get the inactive background name (e.g. `"blue-7"` → `"blue"` → `"dark-blue"`).

New logic: strip the trailing shade suffix, then append `"-dark"` instead (e.g. `"blue-7"` → `"blue"` → `"blue-dark"`).

The lookup into `$colors` uses this derived name. Since `blue-dark` is now a valid entry in `dist/colors.json` (auto-generated by `colors-build` as an alias for `blue-0`), the resolution succeeds.

The generated `dist/projects.zsh` and `dist/projects.json` will have `backgroundInactive.name = "blue-dark"` instead of `"dark-blue"`.

Prior art: `tools/term/zsh/config/theming/projects-build`, `tools/term/zsh/config/theming/__tests__/projects-build.bats`.

## Behavioral Tests

**backgroundInactive name format:**
- A project with `background: "blue-7"` produces `backgroundInactive.name = "blue-dark"` (not `"dark-blue"`)
- A project with `background: "orange-6"` produces `backgroundInactive.name = "orange-dark"`
- `backgroundInactive.ansi` and `backgroundInactive.hex` match the values of `blue-0` / `orange-0` in `dist/colors.json`

**No regression:**
- Projects without a background field still produce no `backgroundInactive` field
- All other project fields (foreground, icon, path, etc.) are unaffected
