## Problem Statement

The shell environment is polluted with 160+ `COLOR_*` and `COLOR_ALIAS_*` exported variables generated by `env-generate-colors`. Every new ZSH session inherits these variables, they appear in `env` output, and they interfere with introspection tools. The current system has no single clean contract between the color definitions and their consumers — each consumer relies on individual exported variables that must all be present and correctly named.

Additionally, non-ZSH consumers (NeoVim Lua) read colors through environment variables at startup, and config-generation scripts (bat, rg, git) use `envsubst` on templates containing `$COLOR_*` placeholders, coupling template syntax to the environment variable naming convention.

## Solution

Replace the `COLOR_*` environment variable system with a ZSH associative array `colors[]` loaded on demand, backed by two generated dist files:

- `dist/colors.zsh` — declares `typeset -gA colors` and populates it with ANSI and hex values for every color and alias
- `dist/colors.json` — a flat JSON map of the same data, for non-ZSH consumers (NeoVim, `projects-build`)

A new `colors-build` script (replacing `env-generate-colors`) generates both dist files from the existing source of truth: `kitty/config/colors.conf` plus the alias definitions embedded in the build script.

A new `colors-template-render` autoload function replaces `envsubst` for config-generation scripts, using a new `{{NAME}}` / `{{NAME:hex}}` placeholder syntax in template files.

At the end of this refactoring, no color-related variables are exported into the shell environment.

## User Stories

1. As a developer, I want to access a color's ANSI value with `$colors[YELLOW_7]`, so that I don't have to remember the `COLOR_` prefix convention.
2. As a developer, I want to access a color's hex value with `$colors[YELLOW_7:hex]`, so that I have a consistent `:hex` suffix pattern across the system.
3. As a developer, I want to access semantic aliases with `$colors[GIT_BRANCH]` and `$colors[GIT_BRANCH:hex]`, so that aliases and raw colors share a single access pattern.
4. As a developer, I want `colors-load-definitions` to be idempotent, so that sourcing it multiple times in a session does not reload the file unnecessarily.
5. As a developer, I want `env` output to be free of `COLOR_*` variables, so that environment introspection is clean and readable.
6. As a developer running `colors-build`, I want both `dist/colors.zsh` and `dist/colors.json` to be regenerated atomically, so that the two dist files are always in sync.
7. As a developer editing `colors.conf` in NeoVim, I want `colors-build` to be triggered on save, so that all downstream consumers are regenerated automatically.
8. As a developer editing `colors-build` itself in NeoVim, I want a re-run to be triggered on save, so that changes to the build script are reflected immediately.
9. As a NeoVim colorscheme, I want to read color hex values from `dist/colors.json` via `F.readJson()`, so that the colorscheme does not depend on environment variables being set at editor startup.
10. As a bat theme template, I want to use `{{BLACK:hex}}` placeholders, so that the template is readable and decoupled from shell variable naming.
11. As a rg config template, I want to use `{{NAME}}` and `{{NAME:hex}}` placeholders, so that color references are explicit and independent of `envsubst`.
12. As a developer running `projects-build`, I want it to read `dist/colors.json` directly without calling `colors-load-definitions`, so that `projects-build` does not depend on shell state.
13. As a developer, I want `colors-template-render <filepath>` to write the rendered output to stdout, so that callers can redirect to any destination.
14. As a developer, I want all ZSH prompt scripts to use `$colors[NAME]` instead of `$COLOR_NAME`, so that prompts work without any exported environment variables.
15. As a developer, I want all tool-configuration scripts (fzf, ls, exa, nnn, zsh-syntax-highlighting, completion) to use `$colors[NAME]`, so that they load correctly after `colors-load-definitions` is called.
16. As a developer, I want all statusbar scripts to load colors via `colors-load-definitions` and use `$colors[NAME]`, so that they are self-contained.
17. As a developer, I want the `colors/` autoload domain to group all color-system functions, so that the domain structure reflects the color system's scope.
18. As a developer, I want `colors-load-definitions` to live in the `colors/` domain (moved from `project/`), so that the function is co-located with the rest of the color system.

## Implementation Decisions

### Source of truth
`kitty/config/colors.conf` remains the source of truth for all color definitions. It is not modified. The alias definitions (semantic names mapped to raw color names) stay embedded in `colors-build`, as they are today.

### `colors[]` array structure
- Raw color ANSI value: `colors[YELLOW_7]` → integer (e.g. `87`)
- Raw color hex value: `colors[YELLOW_7:hex]` → string (e.g. `"#a16207"`)
- Alias ANSI value: `colors[GIT_BRANCH]` → integer (same lookup as raw)
- Alias hex value: `colors[GIT_BRANCH:hex]` → string
- No `ALIAS_` prefix in keys. Aliases and raw colors are indistinguishable from the consumer's perspective.

### `dist/colors.json` structure
```json
{
  "YELLOW_7": { "ansi": 87, "hex": "#a16207" },
  "GIT_BRANCH": { "ansi": 17, "hex": "#d69e2e" }
}
```
Both raw colors and aliases are present at the top level.

### `colors-build` replaces `env-generate-colors`
- Located at `theming/colors-build` (same level as `projects-build`)
- Has a shebang and `set -e` (script, not autoload)
- Generates `dist/colors.zsh` with `typeset -gA colors`
- Generates `dist/colors.json` with the flat map described above
- `env/colors.zsh` is deleted

### `colors-load-definitions` migration
- Moved from `project/` autoload domain to the new `colors/` domain
- Guard changes: checks whether the `colors` associative array is already populated instead of checking `$COLORS_INDEX`
- Sources `dist/colors.zsh` instead of `env/colors.zsh`

### `colors/` autoload domain
New domain grouping:
- `colors-load-definitions` (moved from `project/`)
- `colors-template-render` (new)

### `colors-template-render` autoload function
- Signature: `colors-template-render <filepath>` → stdout
- Calls `colors-load-definitions` internally
- Replaces `{{NAME}}` with `$colors[NAME]` and `{{NAME:hex}}` with `$colors[NAME:hex]`
- Pure ZSH string substitution; no call to `envsubst`

### Template files updated
The three config-generation template files are updated to use `{{NAME}}` / `{{NAME:hex}}` syntax:
- `tools/cli/bat/config/src/oroshi.xml` (was `$COLOR_ALIAS_BLACK_HEXA` etc.)
- `tools/cli/rg/config/src/rgrc.conf`
- `tools/git/git/config/src/gitconfig`

Their corresponding `generate-theme` / `generate-config` scripts call `colors-template-render` instead of running `envsubst`.

### `projects-build` simplification
- Removes the `colors-load-definitions` call at the top
- Removes the `buildColorMap()` function
- Passes `dist/colors.json` directly as `--argjson colors` to `jq`
- The existing tests are updated: the mock of `colors-load-definitions` is replaced by a `dist/colors.json` fixture file in `THEMING_ROOT`

### NeoVim colorscheme migration
`colorscheme/init.lua` is updated to call `F.readJson()` on `dist/colors.json` and build `O.colors.env` from the parsed table, removing all reads of `COLORS_INDEX` and `COLOR_*_HEXA` environment variables.

### NeoVim trigger
`filetypes/colors.lua` gains a new `F.onWrite` trigger: saving `colors-build` re-runs `colors-build`, mirroring the existing `projects-build` trigger pattern.

### `colors-refresh` orchestration order
```
kitty-refresh
→ colors-build          (generates dist/colors.zsh + dist/colors.json)
→ source dist/colors.zsh
→ env-generate-filetypes + source dist/filetypes.zsh
→ projects-build        (reads dist/colors.json)
→ projects-load-definitions
→ generate-theme        (uses colors-template-render)
→ generate-config (rg)  (uses colors-template-render)
→ generate-config (git) (uses colors-template-render)
```

### ZSH consumer migration
All ZSH files that reference `$COLOR_*` or `$COLOR_ALIAS_*` are updated:
- `$COLOR_NAME` → `$colors[NAME]`
- `$COLOR_NAME_HEXA` → `$colors[NAME:hex]`
- `$COLOR_ALIAS_NAME` → `$colors[NAME]`
- `$COLOR_ALIAS_NAME_HEXA` → `$colors[NAME:hex]`
- `colors-load-definitions` is called where needed before first use

Affected areas: prompt scripts, tool-configuration scripts (fzf, ls, exa, nnn, zsh-syntax-highlighting, completion styling), statusbar scripts, AI statusline, colorize helper, git-branch-color.

## Testing Decisions

Good tests verify external behavior only: given a known input, assert the expected output. They do not assert internal implementation (function names, intermediate variables, file structure beyond the public contract).

### `colors-build`
New bats tests in `theming/__tests__/colors-build.bats`:
- Given a minimal `colors.conf`, `dist/colors.zsh` contains the expected `typeset -gA colors` declaration and the correct ANSI/hex entries
- Given a minimal `colors.conf`, `dist/colors.json` contains the correct flat map with `ansi` and `hex` fields
- Aliases are resolved and appear in both output files without the `ALIAS_` prefix
- Both dist files are written atomically (both exist after a single run)

Prior art: `theming/__tests__/projects-build.bats` (same pattern: controlled `THEMING_ROOT`, fixture input, assert output content).

### `colors-template-render`
New bats tests in `colors/__tests__/colors-template-render.bats`:
- A template containing `{{YELLOW_7}}` is rendered with the correct ANSI integer
- A template containing `{{YELLOW_7:hex}}` is rendered with the correct hex string
- An alias placeholder `{{GIT_BRANCH:hex}}` is resolved correctly
- An unknown placeholder is left unchanged (or produces an error — decide at implementation time)

### `colors-load-definitions`
New bats tests in `colors/__tests__/colors-load-definitions.bats`:
- After calling the function, `$colors[YELLOW_7]` returns the expected integer
- Calling the function twice does not re-source the file (idempotency)

### `projects-build` (existing tests updated)
`theming/__tests__/projects-build.bats` is updated to replace the `colors-load-definitions` mock (which set `COLOR_*` env vars) with a `dist/colors.json` fixture file placed in `THEMING_ROOT`. All existing test assertions remain unchanged.

## Out of Scope

- **`env-generate-filetypes` / `dist/filetypes.zsh`**: The filetype system is separate from colors and is not refactored here.
- **`colors.json` for Python/Kitty tab bar**: The Kitty Python tab bar reads ANSI numbers from `dist/projects.json` and converts them via Kitty's internal API — it does not need a `dist/colors.json`.
- **NeoVim Lua test coverage**: Per project policy, no bats wrappers for headless NeoVim; no tests are written for the Lua migration.
- **Renaming color keys**: The color names themselves (`YELLOW_7`, `GIT_BRANCH`, etc.) are not changed.
- **Adding new colors or aliases**: This refactoring does not add, remove, or rename any color or alias.
- **`COLORS_INDEX`**: This variable is eliminated; consumers that used it (NeoVim) are migrated to read from `dist/colors.json` directly.
