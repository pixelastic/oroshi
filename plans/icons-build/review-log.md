## Issue 01 — Migrate source data

### Inconsistent nesting for node, ruby, claude-mcp-context7

```jsonc
"node":                  "",
"node-version-current":  "",
"node-install-in-progress": "",
...
"ruby":                   "",
"ruby-install-in-progress": "",
```

**Problem:** Spec says "organize entries into nested objects by category" — node, ruby, claude-mcp-context7 are left as flat keys rather than nested under `"node"`, `"ruby"` objects.

**Reason skipped:** JSON does not allow a key to be both a scalar string and an object simultaneously. `"node": ""` and `"node": { "version-current": "" }` cannot coexist. These categories have bare keys (the category name itself is an icon key), so flat entries are the only valid representation. The `paths | join("-")` flattening in `colors-build` handles flat keys correctly alongside nested groups.

## Issue 02 — icons-build and loader

### No sort-order test for dist/icons.zsh

```zsh
jq --sort-keys '...' <<<"$srcJson" >"$distJson"
# dist/icons.zsh is generated from dist/icons.json
```

**Problem:** Spec requires sorted keys in `dist/icons.zsh`; no test asserts sort order.
**Reason skipped:** Sort order is structurally guaranteed — `dist/icons.zsh` is generated by piping from `dist/icons.json` which is produced with `jq --sort-keys`. Testing this would duplicate coverage of `jq --sort-keys` behavior, not the build script logic.

### Implicit dependency between JSON and ZSH generation

```zsh
jq --sort-keys '...' <<<"$srcJson" >"$distJson"
# then:
jq -r 'to_entries[] | ...' "$distJson"
```

**Problem:** `dist/icons.zsh` reads `$distJson` which is only valid after JSON generation; dependency is implicit (call order).
**Reason skipped:** Linear script structure makes call order self-evident; extracting an explicit dependency would add complexity with no practical benefit for a two-step build script.
