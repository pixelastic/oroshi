## Problem Statement

Project metadata (colors, icons, paths) is currently defined in a ZSH-specific file (`src/projects-list.zsh`) and compiled into 1274 flat environment variables at shell startup. This approach has three problems: it pollutes every child process's environment with ~1274 variables; it ties project data to ZSH as the authoring format, making it hard to consume from other tools; and the flat variable naming scheme (`PROJECT_ABERLAAS_BACKGROUND_INACTIVE_HEXA`) is verbose and hard to navigate. As the number of projects grows, the system no longer scales cleanly.

## Solution

Make a JSON file the single source of truth for project metadata. A build script reads this source, resolves color names to their ANSI and hex values, and produces two distribution artefacts: an expanded JSON file (for NeoVim and Kitty) and a ZSH associative array file (for ZSH functions). The ZSH array replaces all 1274 flat environment variables. NeoVim and Kitty are migrated to read the new expanded JSON. ZSH functions are migrated in a follow-up PRD.

## User Stories

1. As a developer, I want to define project metadata in a single JSON file, so that I only have one place to add, edit, or remove a project.
2. As a developer, I want the JSON source file to be sorted alphabetically by project name automatically, so that I can find any project by scanning the file without searching.
3. As a developer, I want to define a project's background color using its semantic name (e.g. `"YELLOW_7"`), so that the source file stays readable without knowing terminal color codes by heart.
4. As a developer, I want the build script to expand color names into all three representations (name, ANSI code, hex), so that each consumer can pick the representation it needs without duplicating the resolution logic.
5. As a developer, I want the build script to derive the inactive background color automatically from the active background, so that I never have to write it manually.
6. As a developer, I want paths stored with tilde notation in the source, so that the file stays portable and readable across machines.
7. As a developer, I want the build script to normalise trailing slashes on paths, so that I do not need to remember to add them when editing.
8. As a developer, I want `hideNameInPrompt` to be optional (false by default), so that I only write it in the source file for the rare projects that need it.
9. As a developer, I want the build to produce a ZSH associative array with dot-notation keys mirroring the JSON path, so that reading `PROJECTS[oroshi.background.ansi]` in ZSH is directly analogous to `jq '.oroshi.background.ansi'` on the JSON.
10. As a developer, I want the distribution files to be regenerated automatically when I save the source JSON in NeoVim, so that the rest of the system reflects my changes immediately.
11. As a developer editing via an agent (Claude), I want the distribution files to be regenerated by a git pre-commit hook, so that the dist is always up to date before a commit regardless of editor.
12. As a developer starting a new shell, I want the ZSH array to be regenerated automatically if the dist file is missing, so that the system works after a fresh clone without any manual setup.
13. As a NeoVim user, I want NeoVim to read project metadata from the expanded JSON file, so that it no longer depends on environment variables that may not be set.
14. As a Kitty user, I want Kitty to read project metadata from the expanded JSON file in nested format, so that it uses the same source as all other consumers.
15. As a developer, I want a one-shot migration script that converts the existing `projects-list.zsh` into the new `src/projects.json`, so that I do not have to re-enter all project data by hand.

## Implementation Decisions

### Source file format (`src/projects.json`)

Human-edited. Keys are project names (kebab-case, e.g. `"aberlaas"`), sorted alphabetically. Each value is an object with the following fields:

- `background` — color name string (e.g. `"YELLOW_7"`)
- `foreground` — color name string
- `icon` — UTF-8 string, may contain Nerd Font private-use codepoints
- `path` — tilde-prefixed path string, trailing slash optional
- `hideNameInPrompt` — boolean, omitted when false

### Distribution files (`dist/`)

The `dist/` folder is gitignored (consistent with the pattern used by `config/git/git/dist/`, `config/cli/bat/dist/`, etc.). It is regenerated on demand.

**`dist/projects.json`** — expanded nested format. Each project key maps to an object with:

- `background` — object with `name`, `ansi` (integer), `hex` (string)
- `backgroundInactive` — same shape, auto-derived: strip numeric suffix from background name, prepend `DARK_`
- `foreground` — object with `name`, `ansi`, `hex`
- `icon` — string (preserved as-is)
- `path` — string with tilde notation and trailing slash normalised
- `hideNameInPrompt` — boolean (always present, defaulting to false)

**`dist/projects.zsh`** — ZSH associative array declaration. Keys use dot-notation to mirror JSON paths exactly (e.g. `PROJECTS[aberlaas.background.ansi]`). Color integers are stored as integers. The array name is `PROJECTS`, declared with `typeset -gA`.

### Build script (`src/projects-build`)

A ZSH script that:
1. Re-sorts `src/projects.json` in-place using `jq --sort-keys`
2. Sources `env/colors.zsh` to resolve color names to ANSI codes and hex values
3. Derives `backgroundInactive` by stripping the numeric suffix from the background color name and prepending `DARK_`
4. Writes `dist/projects.json` and `dist/projects.zsh`

The script replaces `src/env-generate-projects`. It does not write to `env/`.

### Startup regeneration (`theming/index.zsh`)

On ZSH startup: if `dist/projects.zsh` does not exist, run `src/projects-build` first. Then source `dist/projects.zsh`. This replaces the current pattern that sources `env/projects.zsh`.

### NeoVim `F.readJson()` helper

A new helper in the NeoVim functions module. Takes an absolute file path, reads it, and returns a parsed Lua table using `vim.fn.json_decode`. Returns `nil` if the file does not exist.

### NeoVim `M.getProjectData()` migration

The existing function currently calls the `project-key` shell command then reads `PROJECT_*` environment variables. It is replaced with a version that calls `F.readJson()` on `dist/projects.json` once (or on first access per project), caches the result, and reads values from the nested Lua table. The per-project cache in `O.projects` is preserved.

### NeoVim autocommand

A new `BufWritePost` autocmd matching `src/projects.json`. On fire: runs `src/projects-build` asynchronously via `vim.fn.jobstart`. After the job completes, runs `:checktime` so NeoVim reloads any open dist files without prompting.

### Git pre-commit hook

Added to the oroshi hook system. If `src/projects.json` appears in the staged diff, runs `src/projects-build` and stages the resulting dist files before the commit proceeds. This covers the case where an agent edits the source without going through NeoVim.

### Migration script (`src/projects-migrate`)

A one-shot ZSH script that sources `src/projects-list.zsh`, iterates the `PROJECTS` associative array, and writes `src/projects.json` in the new format. Preserves all icon characters exactly as stored in the ZSH array without re-encoding. Run once, then deleted or archived.

### What is not changed in this PRD

`env/projects.zsh`, `env/projects.json`, and `src/projects-list.zsh` remain untouched during this PRD. ZSH project functions (`context-project`, `context-badge`, `context-root`, `project-path`, `project-exists`) still read from `env/` and env vars. The old pipeline (`src/projects-list.zsh` → `env-generate-projects` → `env/projects.zsh`) continues to run in parallel until the ZSH functions PRD migrates the consumers.

## Testing Decisions

Good tests verify the observable output of `src/projects-build` from a given input, not its internal logic. Tests should cover: the shape and values of generated JSON, the content of the generated ZSH array, and edge cases in the build (missing path, missing foreground, hideNameInPrompt true vs omitted).

**`src/projects-build`** is the only module with BATS tests. It is the deep module: it encapsulates color resolution, inactive color derivation, path normalisation, and dual-format output. All other changes in this PRD are shallow wiring changes.

Test cases for `src/projects-build`:

- Given a minimal project (background, foreground, icon, path): dist JSON contains the correct `name`, `ansi`, and `hex` for background and foreground
- Given a background of `"GREEN_8"`: `backgroundInactive` is derived as `"DARK_GREEN"` with correct ansi and hex values
- Given a background with no numeric suffix (e.g. `"GREEN"`): `backgroundInactive` is derived as `"DARK_GREEN"`
- Given a project with `hideNameInPrompt: true`: dist JSON contains `true`; given a project without the key: dist JSON contains `false`
- Given a path without trailing slash: dist normalises it with a trailing slash
- Given a valid src: dist ZSH array contains dot-notation keys matching the JSON structure
- Given a valid src: `src/projects.json` is sorted alphabetically after the build runs

Prior art: `config/term/zsh/functions/autoload/project/__tests__/context-project.bats` — uses `run zsh -c` with inline env var overrides to test autoload functions that depend on project state.

## Out of Scope

- **ZSH project functions** (`context-project`, `context-badge`, `context-root`, `project-path`, `project-exists`, `project-key`): these are migrated to read from `PROJECTS[name.attr]` in a follow-up PRD.
- **Deletion of the old pipeline** (`projects-list.zsh`, `env-generate-projects`, `env/projects.*`): deferred until ZSH functions are migrated.
- **`dist/colors.json`**: no JSON distribution of the color palette is produced in this PRD. The build script sources `env/colors.zsh` directly.
- **Color override for `backgroundInactive`**: the inactive color is always auto-derived. Manual override per project is not supported.
- **NeoVim test infrastructure**: NeoVim has no existing test suite. Adding one is out of scope.

## Further Notes

The icon fields in `src/projects.json` contain Nerd Font private-use-area codepoints that are invisible in most editors and stripped silently by some file-write tools. When editing or generating this file programmatically, use `Edit` (not `Write`) and never re-encode icon strings. The migration script must copy icon values verbatim from the ZSH array without passing them through any string transformation.
